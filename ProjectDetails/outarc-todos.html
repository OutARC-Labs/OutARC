Git Workflow

How you stay in sync without stepping on each other. Follow this exactly — it takes 2 minutes per PR and saves hours of conflict hell.
main
Production-ready code only. Protected. Never push directly.
Only accepts PRs from dev
Must pass lint before merging
Auto-deploys to Vercel on merge
dev
Integration branch. Both persons merge feature branches here.
This is your daily working branch
Merge here at end of each day
Sync point for both persons
feat/a-[feature]
Person A's feature branches. Branch from dev, PR back to dev.
Examples: feat/a-auth, feat/a-dashboard
One branch per major feature
Delete after merge
feat/b-[feature]
Person B's feature branches. Branch from dev, PR back to dev.
Examples: feat/b-prisma, feat/b-playwright
One branch per major feature
Delete after merge
# Standard workflow for every feature (run this every time) # 1. Start fresh from dev git checkout dev git pull origin dev # 2. Create your feature branch git checkout -b feat/a-dashboard # or feat/b-prisma, etc. # 3. Work, commit often (every ~45 min) git add . git commit -m "feat: add dashboard layout with sidebar" # 4. Push and open PR to dev (NOT main) git push origin feat/a-dashboard # GitHub → open PR from feat/a-dashboard → dev # Other person reviews, approves, merges # 5. Delete branch after merge git branch -d feat/a-dashboard
Commit message format: feat: [what you added] · fix: [what you fixed] · chore: [setup/config] · wip: [incomplete, don't review]. Keep them short and specific. "fix stuff" is not acceptable.

Playwright + Worker Integration
🎭 Playwright — Handshake Automation
Install Playwright and browser binaries
npm install playwright then npx playwright install chromium. Install inside the worker/ directory scope. Use Chromium only — no need for Firefox/WebKit.
Handshake login function with session cookie reuse
Create worker/handshake-auth.ts. First run: launch headed browser, log in with user credentials, save cookies to /tmp/hs-session-[userId].json. Subsequent runs: load saved cookies, check if session is still valid by hitting Handshake's home page. If invalid (redirected to login), re-authenticate.
key anti-detection measure
Anti-detection: randomized delays utility
Create worker/utils/delays.ts. Export: humanDelay(min=800, max=3000) — random ms delay with slight gaussian distribution (more likely middle values). pageLoadDelay() — 2–5s. typingDelay(char) — 50–150ms per character. Use these everywhere instead of hardcoded waits.
Handshake job scraper
Create worker/handshake-scrape.ts. Navigate to Handshake jobs page with user's filter keywords as query params. Extract from each listing: job title, company name, Handshake job URL, job description text. Return as array of job objects. Limit to first 10 results per run.
Filter already-applied jobs
Before queuing a job for application, check the DB: does an Application row exist with this handshakeUrl for this user? If yes, skip it. This prevents re-applying to the same job across multiple runs.
Handshake application submission function
Create worker/handshake-apply.ts. Navigate to job URL. Click "Apply". Handle multi-step apply forms: some require cover letter upload (create a PDF from the generated text), some have a text field (paste it). Attach resume PDF from /public/resumes/[userId]/resume.pdf. Click submit. Detect success confirmation page.
hardest part of the project
CAPTCHA detection and graceful failure
After page navigation, check for common CAPTCHA indicators: page.locator('[id*="captcha"]').count() > 0 or challenge frame iframes. If detected: mark job as FAILED with reason "captcha_detected". Send Discord notification: "⚠️ CAPTCHA hit on [Company] — needs manual apply." Continue queue.
Screenshot on failure for debugging
Wrap every Playwright action in try/catch. On any unexpected error: take a screenshot with page.screenshot({ path: /tmp/hs-error-[timestamp].png }). Log the screenshot path and the error to console. Don't crash the whole worker — catch and continue to next job.
🔗 Full Worker Pipeline Integration
Wire full pipeline in worker/index.ts
BullMQ Worker processes each job in this order: (1) Fetch user + resume + filter from DB. (2) Decrypt Handshake credentials. (3) Authenticate with Handshake. (4) Scrape job listings matching filter. (5) For each job: call tailorApplication(), then applyToJob(), then saveToHistory(), then notifyDiscord(). (6) Respect 5 apps/hour rate limit.
integration point
Rate limiting: max 5 applications per hour
Before applying to each job, check: how many Applications with appliedAt > (now - 1 hour) exist for this user? If count ≥ 5, use BullMQ's moveToDelayed to re-queue the job for 1 hour later. Log this as "Rate limit hit — re-queued."
Save application to DB after success
After successful Playwright submission: create Application row with status=APPLIED, store coverLetterUsed (full text), store resumeBulletsUsed (JSON), set appliedAt=now. Update Application status to FAILED if Playwright throws.
Discord notification sender
Create worker/notify.ts. Function notifyApplied(job) POSTs to Discord webhook URL with embed: green color, title "✅ Applied", fields for Company, Role, Link, and which cover letter keywords were used. Function notifyFailed(job, reason) for failures with red color.
End-to-end test: one real application
Find a real Handshake posting. Manually trigger one job through the queue. Watch the full pipeline execute in real-time: scrape → tailor → apply → save → Discord notification. This is the Day 2 milestone. Everything else is polish.

🛡️ Stability + Production Hardening
BullMQ retry + backoff configuration
When adding jobs to queue, set: attempts: 3, backoff: { type: 'exponential', delay: 5000 }. This auto-retries failed jobs 3 times with 5s → 10s → 20s delays. After 3 failures, job moves to failed state and Discord notification fires.
Worker crash recovery
Add process-level error handler: process.on('uncaughtException', handler) and process.on('unhandledRejection', handler). Log the error and restart the worker process. On Railway, use a restart policy: always restart on crash with 5s delay.
Input sanitization on all API routes
Every API route that accepts user input should validate with Zod schemas. Especially: filter keywords (max 20 items, each max 50 chars), cover letter text shouldn't be blindly stored without length check. Prevents DB overflow.
Verify credentials are never logged
Search entire codebase for any console.log that could print user passwords, session tokens, or decrypted credentials. grep -r "handshakeCredEnc\|password\|decrypt" worker/ src/ and review each result.

1
User clicks "Start Applying" on dashboard
Frontend calls POST /api/jobs/trigger. API reads user's active JobFilter from DB. Validates user has: uploaded resume, saved Handshake creds, at least one keyword. If missing, returns 400 with specific error message.
PERSON A (API route)
2
Job is added to BullMQ queue
API creates a queue job with: userId, filter settings, timestamp. Returns jobId to frontend. Frontend uses jobId to poll status. Queue job waits in Redis until the worker picks it up.
PERSON B (queue)
3
Worker picks up the job
BullMQ worker receives the job from Redis. Fetches full user record from Postgres including encrypted Handshake credentials. Decrypts credentials using AES-256 key from env.
PERSON B (worker)
4
Playwright authenticates with Handshake
Launches Chromium in headed mode. Loads saved session cookies. Checks if session is valid. If not: navigates to login page, fills in credentials with human-like typing delays, submits form, saves new cookies.
PERSON B (Playwright)
5
Playwright scrapes matching job listings
Navigates to Handshake jobs with user's keyword filters. Extracts: job title, company, URL, description for each listing. Cross-references DB to skip already-applied jobs. Returns up to 10 new job objects.
PERSON B (Playwright)
6
OpenClaw tailors each application
For each job: calls the tailor-application OpenClaw skill with job description + user's parsed resume text. MiniMax-M1 generates a targeted cover letter and rewrites up to 3 resume bullets. Output is validated as JSON.
PERSON A (OpenClaw skill)
7
Playwright submits the application
Navigates to job posting URL. Clicks Apply. Handles the application form: attaches resume PDF, fills cover letter field (or uploads PDF of cover letter), adds any additional required fields. Submits with randomized delays between each action.
PERSON B (Playwright)
8
Application saved to database
Creates Application row in Postgres: status=APPLIED, stores full cover letter text, stores resume bullet rewrites as JSON, records timestamp. Updates dashboard stats (applications today counter increments).
PERSON B (worker)
9
Discord notification fires
Sends embedded message to Discord webhook: green color, "✅ Applied", job title, company, link, cover letter keywords used. User sees it immediately on their phone if Discord is installed.
PERSON B (worker)
10
Dashboard updates in real time
Frontend polling detects new APPLIED entry in /api/jobs/status response. Stats cards update. Toast notification appears: "✅ Applied to [Job] at [Company]." History table shows new row.
PERSON A (frontend polling)
